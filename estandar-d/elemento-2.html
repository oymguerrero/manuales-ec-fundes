Elemento 2: Construir. Esta es la fase de desarrollo técnico — configuras el entorno, programas la solución, la pruebas y la documentas.

El Elemento 2 produce 4 documentos y artefactos: el entorno de desarrollo configurado, la solución desarrollada, el reporte de pruebas técnicas y la documentación técnica.

En el caso de El Surtido, aquí es donde la solución de captura de pedidos, optimización de rutas y predicción de demanda cobra vida en código real que funciona.

3.1 Qué harás en este elemento 4 grandes tareas · desarrollo técnico · 3 min
  1. Configurar el entorno de desarrollo: repositorio con estructura de carpetas organizada según el stack tecnológico, archivo de dependencias con versiones especificadas, README con instrucciones de instalación y ejecución, variables de entorno y credenciales de APIs de IA configuradas de forma segura.
  2. Desarrollar la solución: backend con la lógica de negocio, integración con los modelos/servicios de IA seleccionados, interfaz de usuario funcional y responsiva, esquema de base de datos con respaldos automatizados, medidas de seguridad (validación de datos, autenticación, cifrado, control de acceso basado en roles) y protección de datos personales.
  3. Realizar pruebas técnicas: pruebas unitarias, de integración y funcionales; verificación de vulnerabilidades de seguridad; plan de contingencia ante interrupciones; validación de la calidad de las respuestas de los componentes de IA.
  4. Elaborar la documentación técnica: arquitectura implementada con diagrama actualizado, referencia de endpoints de la API, instrucciones de mantenimiento, decisiones técnicas tomadas con su justificación, configuración de los componentes de IA y dependencias externas.
El Surtido · La solución toma forma en código

Con la propuesta autorizada por Raúl, ahora construyes la solución: una aplicación web que permite a los clientes de El Surtido capturar pedidos por formulario (en lugar de llamada), un módulo de optimización de rutas que usa una API de ruteo con IA para ordenar las paradas de cada camioneta, y un modelo de predicción de demanda entrenado con el historial de pedidos de los últimos dos años.

Todo esto requiere un repositorio organizado, un backend que integre los tres componentes, una interfaz simple para los choferes y los clientes, y pruebas que demuestren que funciona antes de llevarlo a producción.

Para obtener la competencia en este elemento, tu solución debe estar funcionando en el entorno productivo de la MiPyME. Producto 2.1 · El entorno de desarrollo de la solución de transformación digital configurado

El Surtido · Producto 2.1
El repositorio de El Surtido tiene carpetas separadas para backend (API Node.js), frontend (React), módulo de ruteo (Python con biblioteca de optimización) y módulo de predicción (Jupyter notebooks + modelo exportado). El README explica cómo instalar dependencias y configurar las variables de entorno con las claves de la API de ruteo y el modelo de predicción, sin que esas claves queden en el código.
  • (a) Contiene el repositorio de la solución de transformación digital con la estructura de carpetas organizada según el stack tecnológico seleccionado.
  • (b) Incluye el archivo de dependencias con las versiones especificadas de cada librería y framework utilizado.
  • (c) Tiene las instrucciones de instalación, configuración y ejecución del proyecto en el archivo README.
  • (d) Indica la configuración de variables de entorno y credenciales de APIs de IA de forma segura, separadas de la solución de transformación digital.
  • (e) Refiere el stack tecnológico seleccionado correspondiente con la arquitectura definida en la propuesta autorizada.
  • (f) Describe el historial de cambios documentado.

Producto 2.2 · La solución de transformación digital con IA desarrollada

El Surtido · Producto 2.2
La solución de El Surtido tiene: un backend que expone endpoints para captura de pedidos, consulta de rutas y predicción de demanda; la integración con la API de ruteo (con manejo de errores y respuestas de respaldo cuando la API no está disponible); una interfaz web simple donde los clientes capturan pedidos y los choferes ven su ruta del día; y la base de datos con respaldos automatizados cada noche.
  • (a) Contiene el backend operativo con la lógica de negocio implementada según los requerimientos funcionales de la propuesta autorizada.
  • (b) Incluye la integración con los modelos/servicios de IA seleccionados, funcionando con manejo de errores y respuestas de respaldo.
  • (c) Tiene la interfaz de usuario de manera funcional, responsiva, accesible y conectada al backend.
  • (d) Indica el esquema de base de datos implementado con procedimientos de respaldo automatizado configurados.
  • (e) Establece las medidas de seguridad aplicadas sobre validación y sanitización de datos de entrada, autenticación de usuarios, cifrado de datos sensibles y control de acceso basado en roles.
  • (f) Refiere la manera en que se protegen los datos personales en el desarrollo de la solución.
  • (g) Incluye su documentación, nomenclatura consistente y la separación de responsabilidades específicas de las funciones de cada componente del sistema.

Producto 2.3 · El reporte de pruebas técnicas de la solución de transformación digital elaborado

El Surtido · Producto 2.3
El reporte de pruebas de El Surtido documenta: pruebas unitarias para el módulo de captura de pedidos y el módulo de predicción, pruebas de integración entre el backend y la API de ruteo, verificación de que el sistema maneja correctamente una pedido duplicado (escenario real frecuente), y validación de que las respuestas del modelo de predicción tienen una precisión aceptable comparada con los datos históricos reales.
  • (a) Contiene los casos de prueba definidos para cada componente de la solución con datos de entrada y resultados esperados.
  • (b) Incluye los resultados de las pruebas unitarias, las de integración y las funcionales ejecutadas.
  • (c) Apunta las vulnerabilidades de seguridad identificadas, así como el estatus de corrección de las que son solucionables.
  • (d) Tiene la información sobre la verificación realizada del correcto funcionamiento de las integraciones con los modelos/servicios de IA.
  • (e) Describe el plan de contingencia ante la interrupción/saturación del servicio de la solución.
  • (f) Refiere la validación realizada sobre la calidad y relevancia de las respuestas generadas por los componentes de IA.
  • (g) Señala la fecha de las pruebas realizadas, el nombre del responsable de aplicarlas y la versión de la solución de transformación digital evaluada.

Producto 2.4 · La documentación técnica de la solución de transformación digital elaborada

El Surtido · Producto 2.4
La documentación técnica de El Surtido incluye el diagrama de arquitectura actualizado (que refleja la decisión de usar serverless para el módulo de predicción en lugar del servidor propio que se planificó inicialmente), la referencia completa de los endpoints de la API con ejemplos de petición y respuesta, y la justificación técnica de esa decisión de arquitectura con sus ventajas de costo para una empresa del tamaño de El Surtido.
  • (a) Contiene la descripción de la arquitectura implementada con diagrama actualizado de componentes y flujos de datos.
  • (b) Incluye la referencia de los endpoints de la API con sus parámetros, respuestas esperadas y códigos de error.
  • (c) Tiene las instrucciones para mantenimiento, actualización y modificación futura.
  • (d) Incluye las decisiones técnicas tomadas durante el desarrollo con su justificación.
  • (e) Refiere la configuración de los componentes de IA, los modelos utilizados, los prompts implementados con sus parámetros y criterios de selección.
  • (f) Considera las dependencias externas con sus versiones, licencias y procedimiento de actualización.

3.5 AHV — Responsabilidad Actitud observable · penalización si falta · 3 min
AHV · Responsabilidad — Definición conforme al F21

Responsabilidad: La manera en que la solución de transformación digital con IA desarrollada y funcional cumple con los estándares de calidad requeridos: código funcional documentado y alineado a la propuesta autorizada por la MiPyME, cumpliendo con la normatividad de protección de datos personales aplicable.

Esta actitud se observa durante la sesión de evaluación. Su ausencia penaliza el resultado. No se evalúa lo que declaras sobre la calidad de tu código — se evalúa si tu solución efectivamente funciona, está documentada y protege los datos como el F21 y la ley exigen.

La Responsabilidad en el Elemento 2 se manifiesta en la coherencia entre lo que prometiste en la propuesta autorizada (Producto 1.4) y lo que entregaste en la solución (Producto 2.2): si el F21 requiere cifrado de datos sensibles y la propuesta lo incluyó, tu solución debe implementarlo. Si no, la brecha entre promesa y entrega se observa como falta de responsabilidad.

3.6 Glosario del Elemento 2 20 términos del F21 · 8 min
Término Definición conforme al F21
Archivo README Se refiere al documento incluido en un repositorio de código que proporciona información esencial sobre el proyecto, como su propósito, instrucciones de instalación, uso, dependencias, configuración y otros detalles relevantes para usuarios y desarrolladores.
Backend Se refiere a la capa de una aplicación que contiene la lógica de negocio, el procesamiento de datos y la comunicación con bases de datos y servicios externos, no visible para el usuario final.
Cifrado de datos Se refiere al proceso de transformar información legible en un formato codificado que solo puede ser descifrado por personas o sistemas autorizados, protegiendo la confidencialidad de los datos.
Control de acceso basado en roles Se refiere al mecanismo de seguridad que restringe el acceso a los recursos del sistema según el rol asignado a cada usuario dentro de la organización.
Control de versiones Se refiere al sistema que registra los cambios realizados al código fuente a lo largo del tiempo, permitiendo rastrear modificaciones, revertir errores y colaborar entre desarrolladores.
Dependencias externas Se refiere al conjunto de librerías, frameworks, servicios/recursos desarrollados por terceros que una aplicación utiliza para funcionar correctamente, evitando desarrollar funcionalidades desde cero y facilitando la integración, el desarrollo y el mantenimiento del sistema.
Endpoint Se refiere a la dirección/punto de acceso específico de una API a través del cual se realizan solicitudes y se reciben respuestas entre sistemas.
Endpoint de la API Se refiere al punto de acceso específico dentro de una API que permite la comunicación entre sistemas, a través del cual se envían solicitudes y se reciben respuestas para consultar, crear, actualizar o eliminar datos dentro de una aplicación.
Framework Se refiere a la estructura de software prediseñada que proporciona componentes reutilizables, convenciones y herramientas para acelerar el desarrollo de aplicaciones.
Frontend Se refiere a la capa de una aplicación con la que el usuario interactúa directamente, incluyendo la interfaz gráfica, los elementos visuales y la experiencia de navegación.
Interfaz de usuarios Se refiere al conjunto de elementos visuales e interactivos — como pantallas, botones, formularios, menús e iconos — mediante los cuales el usuario interactúa con una aplicación/sistema, facilitando la navegación, el uso de funcionalidades y la comunicación con el backend dentro de una solución digital.
Librería Se refiere al conjunto de funciones, módulos o recursos de código reutilizable que pueden ser integrados en una aplicación para resolver tareas específicas sin necesidad de desarrollarlas desde cero, facilitando el desarrollo y promoviendo buenas prácticas.
Nube Se refiere a los servidores a los que se accede a través de Internet, y al software y bases de datos que se ejecutan en los mismos.
Prompt Se refiere a la instrucción o conjunto de instrucciones que se proporcionan a una herramienta de IA generativa para obtener un resultado específico.
Pruebas de integración Se refiere a las verificaciones del correcto funcionamiento entre múltiples componentes/servicios que interactúan entre sí dentro de la solución.
Pruebas unitarias Se refiere a las verificaciones individuales de cada componente o función del código para confirmar que produce los resultados esperados de forma aislada.
Repositorio de código Se refiere al sistema de almacenamiento y control de versiones que permite gestionar, rastrear y colaborar en el desarrollo del código fuente de una solución tecnológica.
Sanitización de datos de entrada Se refiere al proceso de limpiar, filtrar y validar los datos ingresados por el usuario o sistemas externos para eliminar contenido malicioso/inválido, prevenir vulnerabilidades como inyección de código y garantizar la integridad, seguridad y correcto procesamiento de la información dentro de una aplicación.
Solución de transformación digital Se refiere al repositorio integrado y centralizado de tecnologías y componentes de software — como archivos, carpetas, ejecutables, configuraciones, bases de datos y servicios — organizados en un solo entorno digital que permite gestionar, automatizar e innovar procesos mediante la articulación de frontend, backend e integraciones, incorporando prácticas de desarrollo, seguridad y operación para mejorar la eficiencia, la toma de decisiones y la competitividad organizacional.
Stack tecnológico Se refiere al conjunto de lenguajes de programación, frameworks, librerías, bases de datos y herramientas utilizadas para construir una solución de software.
